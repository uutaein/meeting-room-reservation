## 1. 목적

이 문서는 PRD v1.0에서 추가된 다음 정책을 테스트 가능한 요구사항으로 정의한다.

1. 과거 날짜 및 오늘의 이미 시작된 시간 예약 등록 차단
2. 시작 시각이 지난 예약 카드 회색 음영 표시
3. 예약 카드의 원거리 가독성을 위한 비례 확대
4. 대한민국 공휴일을 제외한 영업일 조회
5. 매주 첫 영업일 조회 기준일 자동 전환

이 문서는 요구사항 기준선만 정의한다.

다음 산출물은 이 단계에서 변경하지 않는다.

1. `features/`
2. `features/step_definitions/`
3. `backend/`
4. `frontend/`

## 2. 범위

### 2.1 포함 범위

1. 단건 예약 생성 시 예약 시작 일시 검증
2. 주간 반복 예약 미리보기 및 생성 시 첫 회차 시작 일시 검증
3. 지난 예약 카드 판정과 회색 음영 표시
4. 1080 x 1920 세로 화면의 예약 카드 비례 확대
5. 대한민국 공식 공휴일, 대체공휴일, 임시공휴일 제외
6. 5, 10, 15, 20 영업일 조회
7. 매주 첫 영업일 00:00 기준일 자동 전환
8. 앱 실행 및 화면 복귀 시 누락된 자동 전환 보정
9. 같은 주의 사용자 수동 기준일 유지

### 2.2 제외 범위

1. 과거 또는 이미 시작된 예약의 수정 제한
2. 과거 또는 이미 시작된 예약의 취소 제한
3. 공휴일 예약 생성 금지
4. 회사별 휴일
5. 부서별 휴무일
6. 개인 휴가
7. 외부 공공 API 실시간 호출
8. 공휴일 데이터 자동 다운로드
9. 화면 Feature 작성
10. 코드 구현

## 3. 용어 정의

| 용어 | 정의 |
|---|---|

## 5. 기능 요구사항

### 5.1 과거 예약 등록 제한

- **REQ-RSV-400-start-passed-date** - 과거 날짜의 단건 예약을 생성하면 HTTP 400과 `ERR_RESERVATION_START_PASSED`를 반환하고 예약을 저장하지 않아야 한다.

- **REQ-RSV-400-start-passed-time** - 오늘 날짜의 예약 시작 시간이 현재 분과 같거나 이전이면 HTTP 400과 `ERR_RESERVATION_START_PASSED`를 반환하고 예약을 저장하지 않아야 한다.

- **REQ-RSV-201-start-future-minute** - 오늘 날짜의 예약 시작 시간이 현재 분보다 1분 이상 늦고 다른 예약 정책을 만족하면 예약을 생성해야 한다.

- **REQ-RSV-201-start-future-date** - 미래 날짜의 예약은 운영시간, 최대 시간, 중복 등 기존 정책을 만족하면 생성해야 한다.

- **REQ-RSV-400-start-server-validation** - 프론트엔드 검증을 우회한 요청에도 백엔드는 동일한 시작 일시 검증을 수행해야 한다.

- **REQ-REC-400-start-passed-preview** - 주간 반복 예약의 첫 회차 시작 일시가 현재 분과 같거나 이전이면 미리보기 요청은 `ERR_RESERVATION_START_PASSED`로 실패해야 한다.

- **REQ-REC-400-start-passed-create** - 주간 반복 예약의 첫 회차 시작 일시가 현재 분과 같거나 이전이면 생성 요청 전체가 실패하고 어떤 회차도 저장되지 않아야 한다.

예약 시작 일시 비교 예시:

| 현재 시각 | 요청 시작 일시 | 결과 |
|---|---|---|
| 2026-06-12 09:30:45 | 2026-06-11 10:00 | 실패 |
| 2026-06-12 09:30:45 | 2026-06-12 09:30 | 실패 |
| 2026-06-12 09:30:45 | 2026-06-12 09:31 | 성공 가능 |
| 2026-06-12 09:30:45 | 2026-06-13 08:00 | 성공 가능 |

### 5.2 대한민국 영업일 조회

- **REQ-BIZDAY-200-calculate** - 기준일부터 지정한 수의 영업일을 계산해야 한다.

- **REQ-BIZDAY-200-count-options** - 지원하는 조회 개수는 5, 10, 15, 20 영업일이다.

- **REQ-BIZDAY-200-exclude-weekend** - 토요일과 일요일은 날짜 카드에서 제외해야 한다.

- **REQ-BIZDAY-200-exclude-holiday** - 대한민국 공식 공휴일은 날짜 카드에서 제외해야 한다.

- **REQ-BIZDAY-200-exclude-substitute** - 대체공휴일은 날짜 카드에서 제외해야 한다.

- **REQ-BIZDAY-200-exclude-temporary** - 정부 지정 임시공휴일은 날짜 카드에서 제외해야 한다.

- **REQ-BIZDAY-200-preserve-count** - 주말과 공휴일을 제외한 뒤에도 사용자가 선택한 개수만큼 날짜 카드를 표시해야 한다.

- **REQ-BIZDAY-400-unsupported-year** - 조회 계산에 지원되지 않는 연도가 필요하면 계산을 중단하고 `ERR_HOLIDAY_DATA_UNAVAILABLE` 상태를 표시해야 한다.

- **REQ-BIZDAY-400-no-silent-fallback** - 지원되지 않는 연도의 공휴일을 일반 영업일로 간주해 날짜 카드를 표시하면 안 된다.

영업일 계산 예시:

| 조건 | 기준일 | 조회 개수 | 기대 결과 |
|---|---|---:|---|
| 월요일이 일반 영업일 | 월요일 | 5 | 월~금 5개 |
| 월요일이 공휴일 | 월요일 | 5 | 화요일부터 다음 주 월요일까지 영업일 5개 |
| 목, 금이 공휴일 | 수요일 | 5 | 수요일 포함 후 다음 영업일까지 계속 계산 |

### 5.3 매주 첫 영업일 기준일 자동 전환

- **REQ-DASH-200-rollover-monday** - 월요일이 영업일이면 월요일 00:00에 조회 기준일을 해당 월요일로 변경해야 한다.

- **REQ-DASH-200-rollover-holiday** - 월요일이 공휴일이면 전환하지 않고 해당 주의 첫 영업일 00:00에 기준일을 변경해야 한다.

- **REQ-DASH-200-rollover-consecutive-holidays** - 월요일부터 연속 공휴일이면 연속 공휴일이 끝난 첫 영업일 00:00에 기준일을 변경해야 한다.

- **REQ-DASH-200-rollover-reload** - 기준일 자동 전환 직후 새 기준일로 예약 목록을 다시 조회해야 한다.

- **REQ-DASH-200-rollover-preserve-count** - 자동 전환 시 사용자가 선택한 영업일 조회 개수를 유지해야 한다.

- **REQ-DASH-200-rollover-once** - 같은 주간 식별자에는 자동 전환을 최대 1회만 수행해야 한다.

- **REQ-DASH-200-manual-date-preserved** - 해당 주의 자동 전환 이후 사용자가 기준일을 수동 변경하면 같은 주의 자동 새로고침, 화면 복귀, 재실행이 그 값을 덮어쓰면 안 된다.

- **REQ-DASH-200-next-week-reset** - 다음 주 첫 영업일이 시작되면 이전 주의 수동 기준일을 새 첫 영업일로 변경해야 한다.

- **REQ-DASH-200-startup-catchup** - 앱 실행 시 현재 주 첫 영업일이 이미 시작됐고 해당 주 자동 전환이 수행되지 않았다면 화면 표시 후 5초 이내에 전환해야 한다.

- **REQ-DASH-200-resume-catchup** - 절전 또는 백그라운드 상태에서 화면으로 복귀할 때 누락된 자동 전환이 있으면 5초 이내에 전환해야 한다.

- **REQ-DASH-200-active-rollover** - 앱이 활성 상태이면 첫 영업일 00:00부터 60초 이내에 자동 전환해야 한다.

- **REQ-DASH-400-rollover-holiday-data** - 첫 영업일 계산에 필요한 공휴일 연도가 지원되지 않으면 자동 전환하지 않고 `ERR_HOLIDAY_DATA_UNAVAILABLE` 상태를 표시해야 한다.

자동 전환 예시:

| 조건 | 기대 전환 시각 | 기대 기준일 |
|---|---|---|
| 월요일이 영업일 | 월요일 00:00 | 월요일 |
| 월요일 공휴일, 화요일 영업일 | 화요일 00:00 | 화요일 |
| 월, 화 연속 공휴일 | 수요일 00:00 | 수요일 |
| 월요일 08:00 앱 실행, 전환 이력 없음 | 실행 후 5초 이내 | 월요일 |
| 월요일 자동 전환 후 수요일 수동 변경 | 같은 주 재실행 시 변경 안 함 | 사용자가 선택한 날짜 |

## 6. 화면 요구사항

### 6.1 예약 등록 화면

- **REQ-UI-DATE-001-min-date** - 예약일 입력은 대한민국 로컬 오늘 날짜보다 이전 날짜를 선택할 수 없게 표시해야 한다.

- **REQ-UI-DATE-002-passed-time** - 오늘 날짜의 시작 시간이 현재 분과 같거나 이전이면 제출 전에 등록 불가 안내를 표시해야 한다.

- **REQ-UI-DATE-003-error-message** - 백엔드가 `ERR_RESERVATION_START_PASSED`를 반환하면 등록 화면을 유지하고 사용자가 입력한 값을 보존해야 한다.

- **REQ-UI-DATE-004-adjust-previous** - 날짜 감소 버튼으로 오늘보다 이전 날짜로 이동할 수 없어야 한다.

- **REQ-UI-DATE-005-recurring** - 주간 반복 예약 선택 상태에서도 동일한 날짜 및 시작 시간 제한을 적용해야 한다.

### 6.2 지난 예약 카드

- **REQ-UI-CARD-001-past-date** - 과거 날짜에 속한 모든 예약 카드는 지난 예약 스타일을 적용해야 한다.

- **REQ-UI-CARD-002-today-passed** - 오늘 날짜의 예약 시작 시간이 현재 분과 같거나 이전이면 지난 예약 스타일을 적용해야 한다.

- **REQ-UI-CARD-003-future** - 예약 시작 일시가 현재 분보다 늦으면 기존 활성 카드 스타일을 유지해야 한다.

- **REQ-UI-CARD-004-gray-style** - 지난 예약 카드는 배경, 테두리, 주요 텍스트를 회색 계열로 표시해야 한다.

- **REQ-UI-CARD-005-readable-contrast** - 지난 예약 카드의 텍스트 대비는 WCAG AA 일반 텍스트 기준인 4.5:1 이상이어야 한다.

- **REQ-UI-CARD-006-not-color-only** - 지난 예약은 색상 외에 접근성 이름 또는 보조 텍스트로 `지난 예약` 상태를 제공해야 한다.

- **REQ-UI-CARD-007-actions-unchanged** - 회색 음영 적용만으로 수정 또는 취소 버튼을 비활성화하면 안 된다.

- **REQ-UI-CARD-008-time-refresh** - 활성 화면에서는 현재 분 변경 후 60초 이내에 카드의 지난 예약 판정을 갱신해야 한다.

- **REQ-UI-CARD-009-resume-refresh** - 화면 복귀 시 5초 이내에 카드의 지난 예약 판정을 다시 계산해야 한다.

### 6.3 예약 카드 비례 확대

1080 x 1920 세로 피봇 화면에서 브라우저 확대율 100%를 기준으로 한다.

| 요소 | 계산된 CSS 크기 | 허용 기준 |
|---|---:|---|
| 회의 목적 | 27px | 26~28px |
| 시간 | 27px | 26~28px |
| 회의실명 | 23px | 22~24px |
| 예약자, 연락처 | 18px | 17~19px |
| 수정, 취소 버튼 | 18px | 17~19px |
| 카드 최소 높이 | 107px | 104~110px |

- **REQ-UI-SCALE-001-proportional** - 카드 내부 글자, 아이콘, 패딩, 열 너비, 버튼을 기존 대비 약 1.67배 비율로 확대해야 한다.

### 6.4 영업일 및 자동 전환 화면

- **REQ-UI-BIZDAY-001-count** - 화면에는 공휴일 제외 후 선택한 개수만큼 날짜 카드를 표시해야 한다.

- **REQ-UI-ROLLOVER-002-list-reload** - 기준 날짜 입력값 변경과 예약 목록 재조회 결과가 일치해야 한다.

- **REQ-UI-ROLLOVER-003-count-preserved** - 자동 전환 후 영업일 조회 개수 선택값이 유지되어야 한다.

## 7. 데이터 요구사항

### 7.1 공휴일 데이터

공휴일 데이터는 앱 패키지에 포함되는 정적 데이터다.

| 필드 | 타입 | 필수 | 예시 | 설명 |
|---|---|---|---|---|
| date | `YYYY-MM-DD` | Y | `2026-03-02` | 공휴일 날짜 |
| name | 문자열 | Y | `삼일절 대체공휴일` | 사용자 및 운영 확인용 명칭 |
| type | 열거형 | Y | `SUBSTITUTE` | `OFFICIAL`, `SUBSTITUTE`, `TEMPORARY` |
| year | 정수 | Y | `2026` | 지원 연도 확인 |

공휴일 데이터 묶음은 다음 메타데이터를 제공해야 한다.

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| supportedYears | 정수 배열 | Y | 공휴일 계산을 지원하는 연도 |
| updatedAt | ISO 날짜 문자열 | Y | 데이터 갱신일 |
| sourceDescription | 문자열 | Y | 데이터 확인 출처 설명 |

### 7.2 대시보드 기준일 상태

자동 전환과 사용자 수동 선택을 구분하기 위해 다음 상태 개념이 필요하다.

| 상태 | 설명 |
|---|---|
| selectedBaseDate | 현재 조회 기준일 |
| selectedBusinessDayCount | 5, 10, 15, 20 중 선택값 |
| lastAutoRolloverWeek | 마지막 자동 전환을 수행한 주간 식별자 |
| manualSelectionWeek | 사용자가 기준일을 직접 변경한 주간 식별자 |

상태 저장 방식은 구현 단계에서 정하되 다음 동작을 만족해야 한다.

1. 같은 주 앱 재실행 후에도 수동 기준일을 자동 전환으로 덮어쓰지 않는다.
2. 다음 주 첫 영업일에는 새 자동 전환을 수행한다.
3. 영업일 조회 개수는 자동 전환으로 초기화하지 않는다.

## 8. 오류 및 상태 코드

| 코드 | HTTP/표시 상태 | 메시지 | 발생 조건 |
|---|---|---|---|
| ERR_RESERVATION_START_PASSED | HTTP 400 | 이미 지난 날짜나 시간에는 예약할 수 없습니다. | 예약 시작 일시가 현재 분과 같거나 이전 |
| ERR_HOLIDAY_DATA_UNAVAILABLE | 화면 오류 상태 | 해당 연도의 공휴일 데이터가 없어 영업일을 계산할 수 없습니다. | 계산에 필요한 연도가 `supportedYears`에 없음 |

오류 처리 규칙:

1. `ERR_RESERVATION_START_PASSED` 발생 시 예약을 저장하지 않는다.
2. 반복 예약에서 오류가 발생하면 어떤 회차도 저장하지 않는다.
3. `ERR_HOLIDAY_DATA_UNAVAILABLE` 발생 시 잘못된 영업일 목록이나 자동 전환을 수행하지 않는다.
4. 오류 메시지는 사용자에게 표시하고 갱신이 필요한 원인을 숨기지 않는다.

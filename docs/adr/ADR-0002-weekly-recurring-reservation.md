# ADR-0002 — 주간 반복 예약 요건 추가

| 항목 | 내용 |
|---|---|
| 상태 | Proposed |

---

## 1. Context

기존 회의실 예약 현황을 확인해보면, 단건 예약만으로 설명하기 어려운 패턴이 있다.

예를 들어 다음과 같은 형태다.

1. 매주 특정 요일에 반복되는 회의
2. 매주 같은 시간에 반복되는 회의
3. 같은 회의실을 정기적으로 사용하는 회의
4. 제목과 참석 인원이 거의 동일한 정례 회의

현재 시스템은 단건 예약 중심으로 설계되어 있다.

따라서 이런 정례 회의를 등록하려면 사용자가 매주 같은 예약을 반복해서 직접 생성해야 한다. 이 방식은 입력 부담이 크고, 특정 주차 예약 누락이나 시간 오입력 가능성이 있다.

> 실제 기존 예약 사용 패턴을 보면 “매주 무슨 요일 몇 시” 형태의 반복 예약 요건이 필요하다.

---

## 2. Decision

회의실 예약 시스템에 **주간 반복 예약** 요건을 추가한다.

MVP 기준 결정은 다음과 같다.

1. 반복 주기는 `매주`만 지원한다.
2. 반복 요일은 1개 요일만 지원한다.
3. 반복 시간은 동일 시작 시간과 동일 종료 시간을 사용한다.
4. 반복 회의실은 동일 회의실을 사용한다.
5. 반복 종료 조건은 `반복 종료일` 또는 `반복 횟수` 중 하나를 사용한다.
6. 반복 예약도 기존 단건 예약 정책을 따른다.
7. 반복 예약 수정 또는 취소 시에는 반복 예약 제목을 정확히 입력해야 한다.

---

## 3. Rationale

주간 반복 예약을 도입하는 이유는 다음과 같다.

1. 실제 사용 패턴에 정례 회의가 존재한다.
2. 같은 예약을 매주 수동으로 등록하는 것은 불편하다.
3. 수동 반복 입력은 예약 누락과 오입력을 만들 수 있다.
4. 반복 예약은 회의실 운영 현황을 더 정확히 반영한다.
5. 제목 확인 절차를 두면 반복 예약 전체 수정과 취소의 실수를 줄일 수 있다.

---

## 4. Alternatives

### 4.1 단건 예약만 유지

장점은 구현이 단순하다는 점이다.

단점은 사용자가 매주 같은 예약을 반복 입력해야 한다는 점이다.

채택하지 않는다.

### 4.2 모든 반복 규칙 지원

매일, 매주, 격주, 매월, 여러 요일 반복을 모두 지원하는 방식이다.

장점은 기능이 풍부하다는 점이다.

단점은 MVP 범위가 커지고, 충돌 처리와 수정 범위 정책이 복잡해진다는 점이다.

채택하지 않는다.

### 4.3 주간 반복만 우선 지원

매주 1개 요일 반복만 우선 지원하는 방식이다.

장점은 실제 정례 회의 패턴을 대부분 수용하면서도 MVP 복잡도를 낮출 수 있다는 점이다.

채택한다.

---

## 5. Consequences

### 5.1 Positive

1. 정례 회의 등록 편의성이 높아진다.
2. 반복 예약 누락 가능성이 줄어든다.
3. 회의실 예약 현황이 실제 운영 패턴에 가까워진다.
4. 사용자는 반복 예약 생성 전에 예상 생성 건수를 확인할 수 있다.

### 5.2 Risks

1. 반복 예약은 여러 날짜에 영향을 준다.
2. 잘못 수정하거나 취소하면 여러 회차 예약이 동시에 바뀔 수 있다.
3. 일부 날짜에서 기존 예약과 시간이 겹칠 수 있다.
4. 단건 예약보다 사용자 확인 절차가 더 중요하다.

### 5.3 Mitigation

1. 반복 예약 수정 또는 취소 시 제목 확인을 요구한다.
2. 반복 예약 생성 전 예상 생성 건수와 충돌 날짜를 보여준다.
3. 반복 예약도 기존 단건 예약의 시간 겹침 검증을 적용한다.
4. MVP에서는 매주 1개 요일 반복만 지원한다.

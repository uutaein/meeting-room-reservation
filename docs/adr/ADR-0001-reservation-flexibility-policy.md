# ADR-0001 — 예약 편의성 우선 정책

| 항목 | 내용 |
|---|---|
| 상태 | Accepted |
| 작성일 | 2026-06-12 |
| 관련 PRD | `docs/PRD.md` v0.5 |
| 관련 SRS | `docs/SRS.md` v0.3 반영 대상 |

---

## 1. Context

회의실 예약 시스템의 목적은 수기 대시보드보다 빠르고 편한 예약 경험을 제공하는 것이다.

초기 기준에는 다음과 같은 엄격한 정책이 포함되어 있었다.

1. 예약 시간을 30분 단위로 제한한다.
2. 회의실 정원을 초과하면 예약을 실패 처리한다.

하지만 실제 회의는 30분 단위로만 시작하거나 끝나지 않는다. 또한 회의실 정원은 공간 사용의 권고 기준에 가깝고, 실제 업무에서는 권고 정원을 넘어서도 예약이 필요한 경우가 있다.

---

## 2. Decision

> 예약 시간은 1분 단위로 허용한다.
>
> 회의실 정원 초과는 서버 실패 조건으로 보지 않는다.
>
> 정원 초과 시에는 사용자에게 경고를 표시하고, 사용자가 확인하면 예약 등록과 예약 변경을 허용한다.

구체적인 정책은 다음과 같다.

1. 회의실 1 권고 정원은 6명이다.
2. 회의실 2 권고 정원은 12명이다.
3. 회의실 1에 7명 이상 예약하면 경고 대상이다.
4. 회의실 2에 13명 이상 예약하면 경고 대상이다.
5. 경고 후 사용자가 확인하면 예약을 계속 진행한다.
6. 같은 회의실의 ACTIVE 예약 시간 겹침은 계속 실패 조건으로 유지한다.
7. 종료 시간이 시작 시간보다 빠르거나 같은 경우도 실패 조건으로 유지한다.
8. 2시간 초과 예약과 운영시간 범위 위반도 실패 조건으로 유지한다.

---

## 3. Alternatives

### 3.1 30분 단위 유지

장점은 화면 구성이 단순하다는 점이다. 단점은 사용자가 실제 회의 시간을 시스템에 맞춰 왜곡해야 한다는 점이다.

채택하지 않는다.

### 3.2 정원 초과 예약 실패 처리

장점은 권고 정원을 강하게 지킬 수 있다는 점이다. 단점은 필요한 예약까지 막아 사용자가 다시 수기 조율로 돌아갈 수 있다는 점이다.

채택하지 않는다.

### 3.3 1분 단위 + 정원 경고 후 허용

장점은 실제 사용 편의성이 높다는 점이다. 단점은 정원 초과 예약이 늘어날 수 있다는 점이다.

채택한다.

---

## 4. Consequences

### Positive

1. 사용자가 실제 회의 시간에 맞춰 예약할 수 있다.
2. 시스템이 업무 흐름을 과도하게 막지 않는다.
3. 정원 초과 상황을 숨기지 않고 명확히 안내할 수 있다.
4. 수기 대시보드보다 빠른 예약 경험을 제공할 수 있다.

### Risks

1. 정원 초과 예약이 반복될 수 있다.
2. 1분 단위 예약은 30분 슬롯형 화면과 직접 맞지 않을 수 있다.
3. 시간 겹침 검증의 중요성이 커진다.

### Mitigation

1. 시간 겹침은 백엔드와 SQLite에서 계속 검증한다.
2. 정원 초과 경고 문구를 명확히 제공한다.
3. 후속 버전에서 정원 초과 이력 저장과 통계를 검토한다.
4. 화면 표시 단위와 예약 입력 단위를 분리해서 설계한다.

---

## 5. Follow-up

1. `docs/SRS.md`를 v0.3으로 현행화한다.
2. `features/*.feature`의 정원 초과 시나리오를 실패 조건에서 경고 후 성공 조건으로 변경한다.
3. 1분 단위 시간 입력 시나리오를 성공 조건으로 변경한다.
4. 테스트 코드의 오류 코드 검증 기준을 `code` 중심으로 정리한다.

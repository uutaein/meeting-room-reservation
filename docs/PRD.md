> 목적은 수기 대시보드 작성 방식을 줄이고, 직원이 회의실 예약 현황을 직접 확인하고 예약할 수 있게 하는 것이다.

### 0.1 현재 구현 기준

2026-06-10 기준 현재 구현은 다음과 같다.

1. 백엔드는 Node.js, Express, SQLite(`better-sqlite3`)로 구성한다.
2. 프론트엔드는 Vue 3와 Vite로 구성하며 로컬 백엔드 API를 호출한다.
3. 날짜별 `ACTIVE` 예약 목록 조회를 지원한다.
4. 예약 등록, 변경, 취소를 지원한다.
5. 예약 변경은 별도 수정 모달에서 기존 값을 채워 처리한다.
6. 취소는 데이터를 삭제하지 않고 `CANCELLED` 상태로 변경한다.
7. 중복 예약은 백엔드에서 검증한다.
8. 정원은 권고 기준으로만 사용한다.
9. 예약 입력 필드는 회의실, 날짜, 시작·종료 시간, 예약자명, 참석 인원, 회의 목적, 연락처다.

아래 항목은 제품 방향에는 포함되지만 현재 구현 완료 범위에는 포함되지 않는다.

1. 예약 상세 조회 전용 API와 상세 화면
2. 비고(`note`) 필드
3. `08:00~20:00` 30분 슬롯 24개를 표시하는 타임라인형 대시보드
4. 취소 예약을 목록에서 함께 표시하는 상태 비교 화면
5. 압축 해제 후 바로 실행하는 폐쇄망 포터블 패키지
6. 1080 × 1920 세로 피봇 화면 최적화 검증
7. 과거 날짜 또는 이미 시작된 예약의 생성·변경·취소 제한

## 1. 배경

현재 회사 사무실 17층에는 명목상 서고로 불리던 공간과 회의실 공간이 있으며, 실제 업무에서는 회의 또는 협업 공간으로 사용되고 있다.

현재 예약 관리는 별도 시스템이 아니라 **수기 대시보드에 직접 작성하는 방식**으로 이루어진다.

그 결과 다음 문제가 발생한다.

1. 예약 작성이 번거롭다.

   * 사용자가 직접 대시보드에 들어가 예약 내용을 작성해야 한다.
   * 날짜, 시간, 회의실, 예약자, 목적을 매번 수기로 입력해야 한다.
   * 입력 형식이 통일되지 않아 예약 현황을 읽기 어렵다.

2. 예약 현황을 한눈에 보기 어렵다.

   * 오늘 어떤 회의실이 어느 시간대에 사용 중인지 빠르게 파악하기 어렵다.
   * 회의실 1과 회의실 2의 사용 가능 시간을 비교하기 불편하다.
   * 예약이 많은 날에는 빈 시간대를 찾는 데 시간이 걸린다.

3. 중복 예약 가능성이 있다.

   * 동일 회의실에 같은 시간대 예약이 중복으로 작성될 수 있다.
   * 일부 시간이 겹치는 예약을 사람이 직접 확인해야 한다.
   * 취소 또는 변경된 예약이 즉시 반영되지 않으면 혼선이 생긴다.

4. 회의실 정원 기준을 관리하기 어렵다.

   * 회의실 1은 6명 규모, 회의실 2는 12명 규모로 사용해야 한다.
   * 수기 관리에서는 참석 인원이 회의실 정원을 초과해도 사전에 막기 어렵다.
   * 작은 회의실에 많은 인원이 예약되거나, 큰 회의실이 불필요하게 점유될 수 있다.

5. 운영 담당자의 관리 부담이 있다.

   * 예약 오류, 중복 예약, 취소 누락을 사람이 직접 확인해야 한다.
   * 특정 회의실이 얼마나 자주 사용되는지 파악하기 어렵다.
   * 회의실 사용 현황 데이터가 축적되지 않는다.

6. 사내 폐쇄망 환경으로 인해 일반적인 웹 서비스 도입이 어렵다.

   * 외부 인터넷 연결을 전제로 한 SaaS 방식은 사용하기 어렵다.
   * CDN, 외부 API, 외부 인증 서비스에 의존하면 반입 및 운영이 제한될 수 있다.
   * 사내 Windows 환경에서 실행 가능한 단순한 배포 방식이 필요하다.

7. 사내 보안 정책상 일반적인 실행 파일 배포가 제한될 수 있다.

   * `.exe` 실행이 제한될 수 있으므로, 일반 설치형 프로그램보다 JS 기반 포터블 앱 형태가 적합하다.
   * npm 기반 패키지로 필요한 파일을 묶어 반입하고, 사내망 PC에서 실행하는 방식을 검토한다.
   * 실행 방식은 보안 정책을 우회하는 방식이 아니라, 사내 허용 정책 안에서 동작해야 한다.

## 2. 목표

이 제품의 목표는 **17층 회의실 1, 회의실 2의 예약 현황을 한눈에 보여주고, 사용자가 직접 예약·변경·취소할 수 있는 간단한 예약 관리 시스템을 만드는 것**이다.

### 2.1 핵심 목표

1. 직원이 회의실 예약 현황을 직접 확인할 수 있게 한다.
2. 직원이 수기 대시보드 작성 없이 회의실을 셀프로 예약할 수 있게 한다.
3. 동일 회의실의 동일 시간대 중복 예약을 방지한다.
4. 회의실 정원을 초과하는 예약을 방지한다.
5. 예약 변경과 취소를 쉽게 처리할 수 있게 한다.
6. 운영 담당자의 수기 확인과 정리 부담을 줄인다.
7. 향후 회의실 이용률과 시간대별 사용 패턴을 확인할 수 있는 기반을 만든다.
8. Windows 기반 사내 PC에서 실행 가능하도록 한다.
9. 1080p 세로 피봇 모니터 환경에서 예약 현황을 보기 쉽게 제공한다.
10. 폐쇄망 환경에서 외부 인터넷 의존 없이 실행할 수 있도록 한다.
11. JS 기반 포터블 앱 형태로 단순하게 실행할 수 있도록 한다.
12. SQLite를 사용해 단일 PC에서 예약 데이터를 저장하고 관리한다.

### 2.2 제품이 해결하려는 핵심 문제

> 현재 문제의 본질은 “회의실이 없어서”가 아니라, **회의실 예약 상태를 신뢰할 수 있고 빠르게 확인할 수 있는 시스템이 없다는 것**이다.

따라서 1차 제품은 복잡한 기능보다 다음에 집중한다.

1. 지금 어떤 회의실이 비어 있는가?
2. 내가 원하는 시간에 예약할 수 있는가?
3. 다른 예약과 시간이 겹치지 않는가?
4. 참석 인원이 회의실 정원 안에 들어오는가?
5. 예약 변경 또는 취소가 즉시 반영되는가?
6. 폐쇄망 Windows 환경에서 간단히 실행할 수 있는가?
7. 세로형 1080p 화면에서 예약 현황을 한눈에 볼 수 있는가?
8. 단일 PC에서 SQLite 기반으로 예약 데이터를 안정적으로 관리할 수 있는가?

## 3. Non-Goals

이번 1차 범위에서는 아래 기능을 포함하지 않는다.

1. 전사 그룹웨어 실시간 연동
2. Outlook, Google Calendar, Teams 캘린더 자동 연동
3. 참석자 일정 자동 조율
4. 회의실 자동 추천 AI 기능
5. 외부 방문자 출입 신청
6. 회의실 문 잠금 장치 또는 출입 통제 시스템 연동
7. QR 체크인
8. 노쇼 패널티
9. 복잡한 승인 워크플로
10. 반복 예약 기능
11. 모바일 앱 개발
12. 장기 통계 리포트 자동 생성
13. 클라우드 기반 SaaS 운영
14. 외부 인터넷 기반 실시간 업데이트
15. 외부 CDN 의존 화면 구성
16. 대규모 서버 인프라 구축
17. 다중 지점 회의실 통합 관리
18. 관리자 전용 기능
19. 다중 사용자 동시 접속 기능
20. 중앙 서버 기반 예약 관리
21. 사내 인증 시스템 연동

> 1차 목표는 대형 회의실 플랫폼이 아니라, 현재 수기 대시보드 방식을 대체하는 **작고 실용적인 단일 PC용 사내 예약 도구**다.

## 4. 예약 대상 공간

1차 제품에서 관리하는 예약 대상은 다음 2개다.

| 공간    |  정원 | 용도     | 비고                      |
| ----- | --: | ------ | ----------------------- |
| 회의실 1 |  6명 | 소규모 회의 | 명목상 서고로 불리던 공간을 회의실로 관리 |
| 회의실 2 | 12명 | 중형 회의  | 일반 회의실                  |

### 4.1 회의실 1

회의실 1은 최대 6명까지 사용할 수 있는 소규모 회의 공간이다.

주요 사용 목적은 다음과 같다.

1. 소규모 회의
2. 1:1 면담
3. 간단한 업무 협의
4. 6명 이하 참석자의 단기 회의

### 4.2 회의실 2

회의실 2는 최대 12명까지 사용할 수 있는 중형 회의 공간이다.

주요 사용 목적은 다음과 같다.

1. 팀 회의
2. 부서 간 협의
3. 7명 이상 12명 이하 회의
4. 외부 참석자가 포함될 수 있는 회의

## 6. 핵심 사용자 시나리오

### 6.1 예약 현황 확인

일반 직원으로서, 오늘 17층 회의실 1과 회의실 2의 예약 현황을 한눈에 보고 싶다.
그래야 내가 사용할 수 있는 시간대를 빠르게 찾을 수 있다.

### 6.2 회의실 예약

회의 주관자로서, 특정 날짜와 시간에 회의실을 예약하고 싶다.
그래야 수기 대시보드에 직접 작성하지 않고 회의 공간을 확보할 수 있다.

### 6.3 참석 인원에 맞는 회의실 선택

회의 주관자로서, 참석 인원에 맞는 회의실을 선택하고 싶다.
그래야 8명 회의에 6명 정원 회의실을 잘못 예약하는 일을 막을 수 있다.

### 6.4 중복 예약 차단

사용자로서, 이미 예약된 시간대에는 같은 회의실을 다시 예약할 수 없다는 안내를 받고 싶다.
그래야 예약 충돌을 사전에 방지할 수 있다.

### 6.5 예약 변경

예약자로서, 회의 시간이 바뀌었을 때 기존 예약을 수정하고 싶다.
그래야 예약 현황이 실제 회의 일정과 일치하게 유지된다.

### 6.6 예약 취소

예약자로서, 사용하지 않게 된 회의실 예약을 취소하고 싶다.
그래야 다른 직원이 해당 시간대를 사용할 수 있다.

### 6.7 단일 PC 예약 관리

사용자로서, 17층 공용 PC 또는 지정된 단일 PC에서 예약 프로그램을 실행하고 싶다.
그래야 별도 서버 없이 예약 현황을 관리할 수 있다.

### 6.8 폐쇄망 환경 실행

사용자로서, 외부 인터넷이 연결되지 않는 사내 폐쇄망 PC에서도 시스템을 실행하고 싶다.
그래야 별도 외부 서비스 의존 없이 회의실 예약 관리를 운영할 수 있다.

### 6.9 포터블 앱 실행

사용자로서, 복잡한 설치 과정 없이 반입된 패키지를 실행하고 싶다.
그래야 개발 지식이 없어도 예약 시스템을 사용할 수 있다.

### 6.10 세로형 대시보드 조회

사용자로서, 세로로 피봇된 1080p 모니터에서 오늘의 회의실 예약 현황을 확인하고 싶다.
그래야 17층 회의실 현황판처럼 화면을 띄워두고 사용할 수 있다.

사용자는 기존 예약을 변경할 수 있어야 한다.

1. 회의실

4. 종료 시간
5. 참석 인원
6. 예약자명
7. 회의 목적

6. 과거 날짜 또는 이미 시작된 예약의 변경 제한은 현재 구현하지 않으며 정책 확정 후 적용한다.

1차 범위에서는 관리자 전용 취소 기능을 별도로 두지 않는다.

### 7.6 JS 기반 포터블 앱 실행

현재 시스템은 npm 스크립트로 Node.js 백엔드와 Vue 개발 서버를 실행한다.

구체적인 실행 방식은 SRS 및 SDD 단계에서 확정한다.

후보 방식은 다음과 같다.

1. npm 패키지 기반 로컬 실행
2. 로컬 브라우저 기반 화면 실행
3. Node.js 기반 로컬 서버 실행
4. SQLite 파일을 사용하는 단일 PC 앱 구조

### 7.7 SQLite 데이터 저장

시스템은 예약 데이터를 SQLite에 저장한다.

SQLite에는 다음 데이터가 저장될 수 있다.

1. 회의실 정보
2. 예약 정보
3. 예약 상태
4. 등록 일시
5. 수정 일시
6. 취소 여부

SQLite는 단일 PC용 예약 관리에는 적합하지만, 다중 사용자 동시 접속이나 중앙 서버 운영 목적에는 적합하지 않을 수 있다.
따라서 1차 제품은 단일 PC 사용을 전제로 한다.

## 8. 비즈니스 규칙

> 아래 규칙은 제품 정책의 의도다. 정확한 DB 제약, 예외 조건, UI 메시지는 SRS 단계에서 더 명확히 정의한다.

### 8.1 회의실 정원

1. 회의실 1의 정원은 6명이다.
2. 회의실 2의 정원은 12명이다.
3. 예약 시 참석 인원은 필수로 입력한다.
4. 참석 인원과 정원은 권고 기준으로만 사용한다.
5. 참석 인원이 7명 이상이면 회의실 1은 권고 대상에서 벗어난다.
6. 참석 인원이 13명 이상이면 회의실 2도 권고 대상에서 벗어난다.

### 8.2 예약 시간 단위

1. 예약은 1분 단위로 가능하다.
2. 시작 시간은 1분 단위여야 한다.
3. 종료 시간은 1분 단위여야 한다.
4. 예약 가능 예시는 다음과 같다.

   * 09:00~09:30
   * 09:00~10:00
   * 09:30~10:30
   * 10:00~12:00
5. 예약 불가 예시는 다음과 같다.

   * 09:10~10:10
   * 09:15~10:15
   * 10:20~11:00

### 8.3 최대 예약 시간

1. 1회 예약은 최대 2시간까지 가능하다.
2. 2시간을 초과하는 예약은 등록할 수 없다.
3. 예를 들어 09:00~11:00 예약은 가능하다.
4. 예를 들어 09:00~11:30 예약은 불가하다.

### 8.4 중복 예약

1. 같은 회의실은 같은 시간대에 하나의 예약만 허용한다.
2. 일부 시간만 겹쳐도 중복 예약으로 본다.
3. 회의실 1과 회의실 2는 서로 다른 공간이므로 같은 시간대에 각각 예약될 수 있다.
4. 취소된 예약은 중복 판단 대상에서 제외한다.
5. 중복 예약 검증은 애플리케이션 로직과 SQLite 제약 또는 트리거를 함께 사용해 처리한다.

### 8.5 예약 시간

1. 예약에는 시작 시간과 종료 시간이 반드시 있어야 한다.
2. 종료 시간은 시작 시간보다 늦어야 한다.
3. 예약 시간은 1분 이상, 2시간 이하로 제한한다.
4. 과거 날짜 예약 제한은 현재 구현하지 않으며 정책 확정 후 적용한다.
5. 예약 가능한 시간대는 회사 운영 정책에 따른다.

### 8.6 예약 변경

1. 사용자는 기존 예약을 변경할 수 있다.
2. 변경 후 예약도 정원 권고, 1분 단위, 최대 2시간, 중복 예약 규칙을 만족해야 한다.
3. 이미 시작된 예약의 변경 가능 여부는 SRS 단계에서 확정한다.
4. 1차 범위에서는 관리자 전용 변경 기능을 두지 않는다.

### 8.7 예약 취소

1. 사용자는 기존 예약을 취소할 수 있다.
2. 취소된 예약은 다른 사용자가 다시 예약할 수 있어야 한다.
3. 이미 시작된 예약의 취소 가능 여부는 SRS 단계에서 확정한다.
4. 1차 범위에서는 관리자 전용 취소 기능을 두지 않는다.

### 8.8 권한

1. 1차 범위에서는 별도 관리자 기능을 제공하지 않는다.
2. 1차 범위에서는 단일 PC에서 사용하는 단순 예약 도구를 전제로 한다.
3. 일반 사용자는 예약 현황 조회, 예약 등록, 예약 변경, 예약 취소를 수행할 수 있다.
4. 운영자 또는 관리자 권한 분리는 후속 버전에서 검토한다.

## 9. 운영 환경 및 제약사항

### 9.1 Windows 실행 환경

본 시스템은 Windows 환경에서 실행 가능해야 한다.

1. 일반적인 사내 업무용 Windows PC에서 동작하는 것을 전제로 한다.
2. 사내 보안 정책상 일반적인 `.exe` 실행이 제한될 수 있다.
3. 따라서 1차 제품은 JS 기반 포터블 앱 또는 npm 패키지 기반 실행 방식을 우선 검토한다.
4. 실행 시 외부 인터넷 연결을 필수로 요구하지 않아야 한다.
5. 설치형 프로그램보다 압축 해제 후 실행 가능한 형태를 우선한다.

### 9.2 JS 기반 포터블 앱 제약

본 시스템은 JS 기반 포터블 앱 형태를 목표로 한다.

다만 실행 방식에 따라 다음 제약이 존재한다.

1. Node.js 기반으로 실행하려면 실행 PC에 Node.js 런타임이 필요하다.
2. Node.js 런타임을 패키지에 포함하면 사실상 실행 파일이 포함될 수 있다.
3. Electron 기반 앱은 사용성이 좋지만 일반적으로 실행 파일 형태로 패키징된다.
4. 순수 브라우저 기반 앱은 실행이 쉽지만 SQLite 파일을 직접 안정적으로 저장하는 데 제약이 있다.
5. 따라서 사내 보안 정책상 허용 가능한 실행 방식은 SRS 단계에서 확인해야 한다.

### 9.3 npm 패키징

본 시스템은 npm 기반 패키징을 고려한다.

1. 필요한 JS 라이브러리는 폐쇄망 반입 전에 패키지에 포함한다.
2. 실행 시 외부 npm registry 접근을 요구하지 않아야 한다.
3. `node_modules` 또는 번들링된 정적 리소스를 함께 반입할 수 있어야 한다.
4. 폐쇄망 내부에서 `npm install`을 다시 수행하지 않는 방향을 우선한다.
5. 실행 명령은 가능한 한 단순해야 한다.

### 9.4 세로형 1080p 화면 기준

본 시스템의 기본 사용 화면은 1080p 모니터를 세로로 피봇한 환경을 기준으로 설계한다.

1. 주요 기준 해상도는 1080 × 1920 세로형 화면으로 본다.
2. 예약 현황 대시보드는 세로 화면에서 회의실 1, 회의실 2의 일간 예약 현황을 한눈에 볼 수 있어야 한다.
3. 사용자가 스크롤을 과도하게 하지 않아도 오늘의 주요 예약 현황을 확인할 수 있어야 한다.
4. 가로형 일반 모니터에서도 사용은 가능해야 하지만, 1차 최적화 기준은 세로형 1080p 화면이다.
5. 화면 크기에 따라 글자, 예약 카드, 시간대 영역이 지나치게 작아지지 않아야 한다.

### 9.5 폐쇄망 제약

본 시스템은 사내 폐쇄망 환경에서 동작할 수 있어야 한다.

1. 외부 CDN, 외부 API, 외부 인증 서비스에 필수 의존하지 않아야 한다.
2. 필요한 정적 리소스는 실행 패키지에 포함되어야 한다.
3. 프로그램 실행에 필요한 파일은 사전에 하나의 패키지로 묶어 반입할 수 있어야 한다.
4. 배포 이후 인터넷을 통한 자동 업데이트는 1차 범위에 포함하지 않는다.
5. 외부 네트워크 연결이 없어도 기본 예약 기능은 동작해야 한다.

### 9.6 단일 PC 제약

본 시스템은 1차 범위에서 단일 PC 사용을 전제로 한다.

1. 여러 사용자가 각자 PC에서 동시에 접속하는 구조는 1차 범위에 포함하지 않는다.
2. 중앙 서버 기반 예약 관리는 1차 범위에 포함하지 않는다.
3. SQLite DB 파일은 단일 PC 로컬 저장소에 위치하는 것을 전제로 한다.
4. 데이터 백업은 수동 파일 백업 또는 후속 기능으로 검토한다.
5. 단일 PC가 고장 나거나 DB 파일이 손상될 경우 복구 방안이 필요할 수 있다.

## 10. 주요 화면 방향

### 10.1 기본 화면 — 예약 현황 대시보드

기본 화면은 사용자가 가장 먼저 보는 화면이다.

핵심 목적은 **회의실 1과 회의실 2의 예약 현황을 한눈에 보여주는 것**이다.

| 영역     | 설명                        |
| ------ | ------------------------- |
| 상단     | 기준 날짜 선택, 조회 영업일 수 선택       |
| 조회 범위  | 기준일부터 5·10·15·20 영업일 조회      |
| 중앙     | 날짜별 예약 목록 테이블              |
| 예약 행   | 회의실, 시간, 예약자, 참석 인원, 목적 표시 |
| 주요 버튼  | 예약 등록, 수정, 취소               |

세로형 1080p 화면에서는 다음 방향을 우선한다.

1. 오늘 날짜를 기본 조회 기준일로 사용한다.
2. 기본 조회 범위는 10영업일이다.
3. 회의실 1과 회의실 2 예약을 하나의 날짜별 목록에서 확인한다.
4. 예약이 없는 날짜는 빈 상태 메시지를 표시한다.
5. 예약 등록 버튼과 각 예약의 수정·취소 버튼을 제공한다.
6. 30분 슬롯형 시각화와 세로형 화면 최적화는 후속 작업으로 남긴다.

### 10.2 예약 등록 화면

예약 등록 화면에서는 최소 입력으로 빠르게 예약할 수 있어야 한다.

필수 입력값은 다음과 같다.

1. 회의실
2. 날짜
3. 시작 시간

### 10.4 운영자 화면

## 14. 리스크 및 고려사항

### 14.1 사용자가 기존 수기 방식에 머무를 가능성

사용자가 새 시스템을 사용하지 않고 기존 대시보드를 계속 사용할 수 있다.

대응 방향은 다음과 같다.

1. 예약 등록 절차를 기존 수기 작성보다 단순하게 만든다.
2. 기본 화면에서 예약 가능 여부를 즉시 확인할 수 있게 한다.
3. 회의실 예약은 시스템 기준을 원칙으로 운영하도록 한다.

### 14.2 예약 데이터 신뢰성 문제

시스템 예약 현황과 실제 사용 현황이 다르면 신뢰도가 떨어진다.

대응 방향은 다음과 같다.

1. 취소 기능을 쉽게 제공한다.
2. 변경 사항을 즉시 반영한다.
3. 예약 현황 화면을 항상 최신 DB 기준으로 표시한다.

### 14.3 관리자 기능 제외에 따른 오남용 가능성

1차 범위에서는 관리자 기능과 권한 분리를 제외한다.

이 경우 사용자가 다른 사람의 예약을 수정하거나 취소할 수 있는 문제가 생길 수 있다.

대응 방향은 다음과 같다.

1. 1차 제품은 단일 PC용 단순 도구라는 전제를 명확히 한다.
2. 예약 변경/취소 시 확인 절차를 둔다.
3. 변경 이력을 남겨 추후 확인할 수 있게 한다.
4. 권한 분리는 후속 버전에서 검토한다.

### 14.4 정원 초과 예약

회의실 1은 6명, 회의실 2는 12명 정원이므로 정원 초과 예약을 막지 않으면 시스템의 의미가 약해진다.

대응 방향은 다음과 같다.

1. 참석 인원을 필수로 입력받는다.
2. 참석 인원이 정원을 초과하면 예약을 차단한다.
3. 참석 인원 기준으로 선택 가능한 회의실을 안내한다.

### 14.5 중복 예약 판단 기준 모호성

시간이 일부만 겹치는 경우, 경계 시간이 같은 경우의 판단 기준이 필요하다.

예를 들어 다음 상황은 SRS에서 명확히 정의해야 한다.

1. 10:00~11:00 예약이 있을 때 11:00~12:00 예약은 가능한가?
2. 10:00~11:00 예약이 있을 때 10:30~11:30 예약은 차단하는가?
3. 취소된 예약은 중복 판단에서 제외하는가?

중복 예약은 단순한 `UNIQUE` 제약조건만으로는 충분하지 않다.

다음 방식이 필요하다.

1. 애플리케이션에서 저장 전 중복 시간 조회
2. SQLite 트리거로 최종 중복 차단
3. 동일 시작 시간 중복 방지를 위한 보조 인덱스
4. 취소 상태 예약 제외 처리

### 14.6 폐쇄망 반입 제약

폐쇄망에서는 외부 인터넷 연결, 자동 업데이트, 외부 라이브러리 다운로드가 제한될 수 있다.

대응 방향은 다음과 같다.

1. 실행에 필요한 파일을 패키지에 모두 포함한다.
2. 외부 CDN을 사용하지 않는다.
3. 외부 API 의존 기능은 1차 범위에서 제외한다.
4. 업데이트는 새 패키지 반입 방식으로 처리할 수 있게 한다.
5. npm 의존성은 반입 전에 모두 준비한다.

### 14.7 JS 기반 포터블 앱의 실행 제약

JS 기반 포터블 앱도 실행 방식에 따라 런타임이 필요하다.

주의할 점은 다음과 같다.

1. Node.js 실행이 필요하면 `node.exe` 실행 제한에 걸릴 수 있다.
2. Electron은 편리하지만 일반적으로 실행 파일 형태가 된다.
3. 순수 브라우저 앱은 실행이 쉽지만 SQLite 저장에 제약이 있다.
4. 사내 보안 정책에서 어떤 실행 방식이 허용되는지 확인해야 한다.

### 14.8 SQLite 단일 PC 저장 제약

SQLite는 단일 PC 저장에는 적합하지만, 다중 사용자 동시 접속이나 중앙 서버 구조에는 한계가 있다.

대응 방향은 다음과 같다.

1. 1차 제품은 단일 PC 사용으로 범위를 제한한다.
2. DB 파일 위치를 명확히 정한다.
3. 수동 백업 방식을 제공한다.
4. 향후 다중 사용자 환경이 필요하면 중앙 서버 구조로 전환을 검토한다.

### 14.9 세로형 화면 가독성 문제

1080p 모니터를 세로로 피봇하면 높이는 충분하지만 가로 폭은 제한된다.

대응 방향은 다음과 같다.

1. 화면을 가로로 복잡하게 나누지 않는다.
2. 회의실 1, 회의실 2 비교를 단순하게 구성한다.
3. 예약 카드는 짧고 명확하게 표시한다.
4. 상세 정보는 클릭 또는 선택 시 별도 영역에 표시한다.
5. 30분 단위 슬롯이 너무 작아지지 않도록 한다.

## 15. 개발팀에게

이 제품은 복잡한 전사 회의실 플랫폼을 만드는 것이 아니다.

현재 목표는 17층 회의실 1, 회의실 2의 예약을 수기 대시보드로 관리하는 불편을 줄이고, 사용자가 예약 현황을 직접 확인하고 셀프로 예약할 수 있게 하는 것이다.

또한 이 시스템은 일반적인 인터넷 연결 환경이 아니라, **사내 폐쇄망 Windows 환경**에서 실행될 가능성이 높다.
사내 보안 정책상 일반적인 `.exe` 실행이 제한될 수 있으므로, JS 기반 포터블 앱 또는 npm 패키지 기반 실행 방식을 우선 검토한다.

1차 구현에서는 다음 원칙을 우선한다.

1. 예약 현황을 한눈에 볼 수 있어야 한다.
2. 예약 등록이 수기 대시보드보다 쉬워야 한다.
3. 같은 회의실의 중복 예약은 반드시 막아야 한다.
4. 회의실 정원을 초과하는 예약은 막아야 한다.
5. 예약은 1분 단위로 제한한다.
6. 1회 예약은 최대 2시간으로 제한한다.
7. 예약 변경과 취소가 단순해야 한다.
8. Windows 환경에서 실행 가능해야 한다.
9. JS 기반 포터블 앱 또는 npm 기반 패키지 실행 방식을 우선 검토한다.
10. SQLite에 예약 데이터를 저장한다.
11. 단일 PC 사용을 전제로 한다.
12. 관리자 기능은 1차 범위에서 제외한다.
13. 폐쇄망 반입이 가능한 패키지 형태여야 한다.
14. 외부 인터넷 연결 없이 기본 기능이 동작해야 한다.
15. 세로형 1080p 화면에서 가독성이 좋아야 한다.
16. 상세 통계, 알림, 외부 시스템 연동은 후순위로 둔다.

본 제품은 1차 범위에서 단일 PC의 단일 사용자 사용을 전제로 한다.
별도 로그인, 사용자 계정, 관리자 권한, 다중 사용자 동시 접속은 제공하지 않는다.
예약자는 예약 등록 시 본인 이름을 직접 입력한다.
사용자 인증이나 권한 분리는 후속 버전에서 검토한다.

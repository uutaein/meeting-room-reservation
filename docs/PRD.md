# PRD — 17층 서고·회의실 예약 관리 시스템

| 항목 | 내용 |
|---|---|
| 문서 종류 | PRD (Product Requirements Document) |
| 제품명 | 17층 서고·회의실 예약 관리 시스템 |
| 버전 | v0.7 |
| 상태 | 과거 예약 등록 제한 및 지난 예약 표시 정책 반영 |
| 작성일 | 2026-06-12 |
| 대상 공간 | 회의실 1, 회의실 2 |
| 실행 환경 | Windows |
| 실행 형태 | Node.js 로컬 서버 + Vue 웹 화면 |
| 패키징 방식 | npm 기반 패키지 |
| 화면 기준 | 웹 대시보드, 1080p 세로 피봇 최적화는 후속 작업 |
| 배포 환경 | 사내 폐쇄망 |
| 데이터 저장 | SQLite |
| 사용 형태 | 단일 PC용 |
| 관리자 기능 | 1차 범위 제외 |
| 예약 단위 | 1분 단위 |
| 최대 예약 시간 | 2시간 |
| 주요 사용자 | 일반 직원, 회의 주관자 |
| 현재 구현 범위 | 날짜별 예약 조회, 예약 등록·변경·취소, 중복 예약 차단, 정원 권고 및 경고 |

> 이 문서는 17층에 위치한 명목상 서고 공간과 회의실 공간을 실제 업무 기준에 맞춰 **회의실 1, 회의실 2**로 관리하기 위한 예약 시스템 PRD다.
> 목적은 수기 대시보드 작성 방식을 줄이고, 직원이 회의실 예약 현황을 직접 확인하고 예약할 수 있게 하는 것이다.

### 0.1 v0.7 정책 변경 요약

1. 과거 날짜에는 신규 단건 예약과 주간 반복 예약을 등록할 수 없다.
2. 오늘 날짜에는 예약 시작 시간이 현재 시각보다 늦은 경우에만 등록할 수 있다.
3. 예약 시작 일시가 현재 시각과 같거나 이전이면 신규 예약 등록을 차단한다.
4. 등록 가능 여부의 최종 판단은 백엔드 서버의 로컬 날짜와 시각을 기준으로 한다.
5. 프론트엔드는 과거 날짜 선택과 이미 시작된 시간 제출을 사전에 제한하고 사용자에게 이유를 안내한다.
6. 예약 목록에서 시작 일시가 현재 시각과 같거나 이전인 예약 카드는 회색 음영으로 표시한다.
7. 회색 음영은 시간 경과 상태를 알리는 시각적 표현이며, 수정·취소 가능 여부를 변경하지 않는다.
8. 과거 또는 이미 시작된 예약의 수정·취소 제한은 별도 정책으로 확정한다.

### 0.2 현재 구현 기준

2026-06-12 기준 현재 구현은 다음과 같다.

1. 백엔드는 Node.js, Express, SQLite(`better-sqlite3`)로 구성한다.
2. 프론트엔드는 Vue 3와 Vite로 구성하며 로컬 백엔드 API를 호출한다.
3. 날짜별 `ACTIVE` 예약 목록 조회를 지원한다.
4. 예약 등록, 변경, 취소를 지원한다.
5. 예약 변경은 별도 수정 모달에서 기존 값을 채워 처리한다.
6. 취소는 데이터를 삭제하지 않고 `CANCELLED` 상태로 변경한다.
7. 중복 예약은 백엔드에서 검증한다.
8. 정원은 권고 기준으로만 사용하며, 초과 시 경고 후 예약을 허용한다.
9. 예약 입력 필드는 회의실, 날짜, 시작·종료 시간, 예약자명, 참석 인원, 회의 목적, 연락처다.

아래 항목은 제품 방향에는 포함되지만 현재 구현 완료 범위에는 포함하지 않는다.

1. 예약 상세 조회 전용 API와 상세 화면
2. 비고(`note`) 필드
3. 취소 예약을 목록에서 함께 표시하는 상태 비교 화면
4. 압축 해제 후 바로 실행하는 폐쇄망 포터블 패키지
5. 1080 × 1920 세로 피봇 화면 최적화 검증
6. 과거 날짜 또는 이미 시작된 시간의 예약 등록 제한과 지난 예약 카드 음영 표시
7. 정원 초과 경고 이력 저장

## 1. 배경

현재 회사 사무실 17층에는 명목상 서고로 불리던 공간과 회의실 공간이 있으며, 실제 업무에서는 회의 또는 협업 공간으로 사용되고 있다.

현재 예약 관리는 별도 시스템이 아니라 **수기 대시보드에 직접 작성하는 방식**으로 이루어진다.

그 결과 다음 문제가 발생한다.

1. 예약 작성이 번거롭다.
   - 사용자가 직접 대시보드에 들어가 예약 내용을 작성해야 한다.
   - 날짜, 시간, 회의실, 예약자, 목적을 매번 수기로 입력해야 한다.
   - 입력 형식이 통일되지 않아 예약 현황을 읽기 어렵다.
2. 예약 현황을 한눈에 보기 어렵다.
   - 오늘 어떤 회의실이 어느 시간대에 사용 중인지 빠르게 파악하기 어렵다.
   - 회의실 1과 회의실 2의 사용 가능 시간을 비교하기 불편하다.
   - 예약이 많은 날에는 빈 시간대를 찾는 데 시간이 걸린다.
3. 중복 예약 가능성이 있다.
   - 동일 회의실에 같은 시간대 예약이 중복으로 작성될 수 있다.
   - 일부 시간이 겹치는 예약을 사람이 직접 확인해야 한다.
   - 취소 또는 변경된 예약이 즉시 반영되지 않으면 혼선이 생긴다.
4. 회의실 정원 기준을 안내하기 어렵다.
   - 회의실 1은 6명 규모, 회의실 2는 12명 규모로 사용하는 것이 권장된다.
   - 다만 실제 업무에서는 권고 정원을 초과해도 회의실을 사용해야 하는 상황이 생길 수 있다.
   - 따라서 시스템은 정원 초과를 차단하기보다 사용자가 인지하고 선택하도록 안내해야 한다.
5. 운영 담당자의 관리 부담이 있다.
   - 예약 오류, 중복 예약, 취소 누락을 사람이 직접 확인해야 한다.
   - 특정 회의실이 얼마나 자주 사용되는지 파악하기 어렵다.
   - 회의실 사용 현황 데이터가 축적되지 않는다.
6. 사내 폐쇄망 환경으로 인해 일반적인 웹 서비스 도입이 어렵다.
   - 외부 인터넷 연결을 전제로 한 SaaS 방식은 사용하기 어렵다.
   - 외부 API, 외부 인증 서비스에 의존하면 반입 및 운영이 제한될 수 있다.
   - 사내 Windows 환경에서 실행 가능한 단순한 배포 방식이 필요하다.

## 2. 목표

이 제품의 목표는 **17층 회의실 1, 회의실 2의 예약 현황을 한눈에 보여주고, 사용자가 직접 예약·변경·취소할 수 있는 간단한 예약 관리 시스템을 만드는 것**이다.

### 2.1 핵심 목표

1. 직원이 회의실 예약 현황을 직접 확인할 수 있게 한다.
2. 직원이 수기 대시보드 작성 없이 회의실을 셀프로 예약할 수 있게 한다.
3. 동일 회의실의 동일 시간대 중복 예약을 방지한다.
4. 회의실 권고 정원을 초과하는 예약은 차단하지 않고 명확히 경고한다.
5. 예약 변경과 취소를 쉽게 처리할 수 있게 한다.
6. 운영 담당자의 수기 확인과 정리 부담을 줄인다.
7. 향후 회의실 이용률과 시간대별 사용 패턴을 확인할 수 있는 기반을 만든다.
8. Windows 기반 사내 PC에서 실행 가능하도록 한다.
9. 1080p 세로 피봇 모니터 환경에서 예약 현황을 보기 쉽게 제공한다.
10. 폐쇄망 환경에서 외부 인터넷 의존 없이 실행할 수 있도록 한다.
11. JS 기반 포터블 앱 또는 npm 기반 실행 형태로 단순하게 실행할 수 있도록 한다.
12. SQLite를 사용해 단일 PC에서 예약 데이터를 저장하고 관리한다.

### 2.2 제품이 해결하려는 핵심 문제

> 현재 문제의 본질은 “회의실이 없어서”가 아니라, **회의실 예약 상태를 신뢰할 수 있고 빠르게 확인할 수 있는 시스템이 없다는 것**이다.

따라서 1차 제품은 복잡한 기능보다 다음에 집중한다.

1. 지금 어떤 회의실이 비어 있는가?
2. 내가 원하는 시간에 예약할 수 있는가?
3. 다른 예약과 시간이 겹치지 않는가?
4. 참석 인원이 권고 정원을 초과하는지 알 수 있는가?
5. 권고 정원 초과를 인지한 뒤에도 필요한 예약을 진행할 수 있는가?
6. 예약 변경 또는 취소가 즉시 반영되는가?
7. 폐쇄망 Windows 환경에서 간단히 실행할 수 있는가?
8. 세로형 1080p 화면에서 예약 현황을 한눈에 볼 수 있는가?
9. 단일 PC에서 SQLite 기반으로 예약 데이터를 안정적으로 관리할 수 있는가?

## 3. Non-Goals

이번 1차 범위에서는 아래 기능을 포함하지 않는다.

1. 전사 그룹웨어 실시간 연동
2. Outlook, Google Calendar, Teams 캘린더 자동 연동
3. 참석자 일정 자동 조율
4. 회의실 자동 추천 AI 기능
5. 외부 방문자 출입 신청
6. 회의실 문 잠금 장치 또는 출입 통제 시스템 연동
7. QR 체크인
8. 노쇼 패널티
9. 복잡한 승인 워크플로
10. 반복 예약 기능
11. 모바일 앱 개발
12. 장기 통계 리포트 자동 생성
13. 클라우드 기반 SaaS 운영
14. 외부 인터넷 기반 실시간 업데이트
15. 외부 CDN 의존 화면 구성
16. 대규모 서버 인프라 구축
17. 다중 지점 회의실 통합 관리
18. 관리자 전용 기능
19. 다중 사용자 동시 접속 기능
20. 중앙 서버 기반 예약 관리
21. 사내 인증 시스템 연동
22. 정원 초과 예약의 서버 차단

> 1차 목표는 대형 회의실 플랫폼이 아니라, 현재 수기 대시보드 방식을 대체하는 **작고 실용적인 단일 PC용 사내 예약 도구**다.

## 4. 예약 대상 공간

| 공간 | 권고 정원 | 용도 | 비고 |
|---|---:|---|---|
| 회의실 1 | 6명 | 소규모 회의 | 명목상 서고로 불리던 공간을 회의실로 관리 |
| 회의실 2 | 12명 | 중형 회의 | 일반 회의실 |

### 4.1 회의실 1

회의실 1은 권고 정원 6명의 소규모 회의 공간이다.

권고 정원을 초과하는 예약은 시스템에서 차단하지 않는다. 대신 경고 모달을 통해 사용자가 초과 사실을 인지하게 한다.

### 4.2 회의실 2

회의실 2는 권고 정원 12명의 중형 회의 공간이다.

권고 정원을 초과하는 예약은 시스템에서 차단하지 않는다. 대신 경고 모달을 통해 사용자가 초과 사실을 인지하게 한다.

## 5. 핵심 기능

### 5.1 예약 현황 대시보드

기본 화면은 예약 현황 대시보드다.

사용자는 이 화면에서 회의실 1과 회의실 2의 예약 상태를 한눈에 확인할 수 있어야 한다.

필요한 정보는 다음과 같다.

1. 날짜
2. 회의실 목록
3. 시간대별 예약 현황
4. 예약 가능 시간
5. 예약된 시간
6. 회의 제목 또는 이용 목적
7. 예약자
8. 참석 인원
9. 예약 상태
10. 새 예약 등록 버튼
11. 시작 시각이 지난 예약의 시각적 구분

예약 시간 경과 표시는 다음 UX를 따른다.

1. 예약 시작 일시가 현재 시각과 같거나 이전이면 해당 예약 카드를 회색 음영으로 표시한다.
2. 과거 날짜의 예약 카드는 모두 회색 음영으로 표시한다.
3. 오늘 날짜의 예약은 시작 시간이 지나기 전까지 기존 카드 스타일을 유지한다.
4. 회색 음영은 목록에서 지난 예약을 빠르게 구분하기 위한 표현이다.
5. 회색 음영만으로 수정·취소 버튼을 비활성화하지 않는다.

### 5.2 회의실 예약 등록

사용자는 회의실을 선택해 예약을 등록할 수 있어야 한다.

| 항목 | 필수 여부 | 설명 |
|---|---|---|
| 회의실 | 필수 | 회의실 1 또는 회의실 2 |
| 날짜 | 필수 | 예약 날짜 |
| 시작 시간 | 필수 | 회의 시작 시간 |
| 종료 시간 | 필수 | 회의 종료 시간 |
| 예약자 | 필수 | 예약을 등록한 사용자 |
| 참석 인원 | 필수 | 회의 참석 예정 인원 |
| 회의 목적 | 필수 | 회의 또는 이용 목적 |
| 연락처 | 필수 | 예약자 연락처 |

정원 초과 시에는 다음 UX를 따른다.

1. 사용자가 권고 정원보다 많은 참석 인원을 입력한다.
2. 시스템은 경고 모달을 표시한다.
3. 사용자가 취소하면 예약 등록을 중단한다.
4. 사용자가 확인하면 예약 등록을 계속 진행한다.

예약 일시 선택과 등록은 다음 정책을 따른다.

1. 과거 날짜는 선택하거나 제출할 수 없다.
2. 오늘 날짜에는 현재 시각보다 늦은 시작 시간만 등록할 수 있다.
3. 예약 시작 일시가 현재 시각과 정확히 같아도 등록할 수 없다.
4. 미래 날짜는 운영시간, 최대 예약 시간, 중복 예약 등 기존 정책을 만족하면 등록할 수 있다.
5. 프론트엔드는 과거 날짜 선택을 제한하고, 이미 시작된 시간을 제출하면 명확한 안내를 표시한다.
6. 백엔드는 프론트엔드 제한과 관계없이 동일한 정책을 다시 검증한다.
7. 주간 반복 예약은 첫 회차의 시작 일시가 현재 시각보다 늦어야 한다.
8. 등록 차단 시 어떤 단건 예약 또는 반복 예약 회차도 저장하지 않는다.

### 5.3 예약 변경

사용자는 기존 예약을 변경할 수 있어야 한다.

변경 후에는 다음 조건을 따른다.

1. 같은 회의실의 기존 ACTIVE 예약과 시간이 겹치지 않아야 한다.
2. 참석 인원이 회의실 권고 정원을 초과하면 경고를 표시하되, 사용자가 확인하면 변경을 허용한다.
3. 종료 시간은 시작 시간보다 늦어야 한다.
4. 예약 시간은 1분 단위여야 한다.
5. 예약 시간은 최대 2시간을 초과할 수 없다.
6. 과거 날짜 또는 이미 시작된 예약의 변경 제한은 이번 등록 정책에 포함하지 않으며 별도 정책 확정 후 적용한다.

### 5.4 예약 취소

사용자는 기존 예약을 취소할 수 있어야 한다.

취소된 예약은 삭제하지 않고 `CANCELLED` 상태로 변경한다. 취소된 예약 시간대는 다른 사용자가 다시 예약할 수 있어야 한다.

## 6. 비즈니스 규칙

### 6.1 회의실 정원

1. 회의실 1의 권고 정원은 6명이다.
2. 회의실 2의 권고 정원은 12명이다.
3. 예약 시 참석 인원은 필수로 입력한다.
4. 참석 인원과 정원은 강제 차단 기준이 아니라 권고 기준으로 사용한다.
5. 회의실 1 예약에서 참석 인원이 7명 이상이면 정원 경고 대상이다.
6. 회의실 2 예약에서 참석 인원이 13명 이상이면 정원 경고 대상이다.
7. 정원 경고 대상이어도 사용자가 확인하면 예약 등록과 변경을 허용한다.
8. 정원 초과 예약은 운영 리스크로 관리하되, MVP에서는 사용자 편의성을 우선한다.

### 6.2 예약 시간 단위

1. 예약은 1분 단위로 가능하다.
2. 시작 시간은 `HH:mm` 형식으로 입력한다.
3. 종료 시간은 `HH:mm` 형식으로 입력한다.
4. 30분 단위 슬롯은 화면 표현 또는 후속 타임라인 시각화에서만 사용할 수 있으며, 예약 가능 단위를 제한하는 정책으로 사용하지 않는다.

예약 가능 예시는 다음과 같다.

- `09:00~09:30`
- `09:10~10:10`
- `09:15~10:15`
- `10:20~11:00`

### 6.3 최대 예약 시간

1. 1회 예약은 최대 2시간까지 가능하다.
2. 2시간을 초과하는 예약은 등록할 수 없다.
3. 예를 들어 `09:00~11:00` 예약은 가능하다.
4. 예를 들어 `09:00~11:30` 예약은 불가하다.

### 6.4 중복 예약

1. 같은 회의실은 같은 시간대에 하나의 ACTIVE 예약만 허용한다.
2. 일부 시간만 겹쳐도 중복 예약으로 본다.
3. 회의실 1과 회의실 2는 서로 다른 공간이므로 같은 시간대에 각각 예약될 수 있다.
4. 취소된 예약은 중복 판단 대상에서 제외한다.
5. 중복 예약 검증은 애플리케이션 로직과 SQLite 트리거를 함께 사용해 처리한다.

### 6.5 과거 예약 등록 제한

1. 시스템의 로컬 날짜와 시각을 예약 가능 여부의 기준으로 사용한다.
2. 예약 시작 일시는 `예약 날짜 + 시작 시간`으로 판단한다.
3. 예약 시작 일시가 현재 시각보다 늦어야 신규 예약을 등록할 수 있다.
4. 예약 시작 일시가 현재 시각과 같거나 이전이면 등록을 차단한다.
5. 단건 예약과 주간 반복 예약에 동일한 기준을 적용한다.
6. 주간 반복 예약의 시작 회차가 등록 불가능하면 반복 예약 전체를 생성하지 않는다.
7. 백엔드를 최종 검증 주체로 사용한다.
8. 날짜 및 시각 비교는 서버가 실행 중인 PC의 로컬 시간대를 기준으로 한다.

### 6.6 지난 예약 카드 표시

1. 목록의 예약 시작 일시가 현재 시각과 같거나 이전이면 지난 예약으로 표시한다.
2. 지난 예약 카드는 배경, 테두리, 텍스트를 회색 계열 음영으로 표시한다.
3. 텍스트를 읽을 수 있는 대비를 유지해야 한다.
4. 색상만으로 상태를 구분하지 않도록 접근 가능한 상태 설명을 제공하는 방안을 화면 명세에서 구체화한다.
5. 카드 음영 표시는 데이터 상태를 변경하지 않는다.
6. 수정·취소 가능 여부는 별도 정책이며 카드 음영과 결합하지 않는다.

## 7. 운영 환경 및 제약사항

1. 본 시스템은 Windows 환경에서 실행 가능해야 한다.
2. 외부 인터넷 연결 없이 기본 예약 기능이 동작해야 한다.
3. 1차 범위에서는 단일 PC 사용을 전제로 한다.
4. SQLite DB 파일은 단일 PC 로컬 저장소에 위치하는 것을 전제로 한다.
5. 데이터 백업은 수동 파일 백업 또는 후속 기능으로 검토한다.

## 11. 리스크 및 고려사항

### 11.1 정원 초과 예약 리스크

회의실 권고 정원을 초과하는 예약을 허용하면 공간 부족이나 회의 품질 저하가 발생할 수 있다.

대응 방향은 다음과 같다.

1. 참석 인원을 필수로 입력받는다.
2. 참석 인원이 권고 정원을 초과하면 경고 모달을 표시한다.
3. 더 큰 회의실 선택을 권장한다.
4. 사용자가 확인하면 예약을 허용한다.
5. 후속 버전에서 정원 초과 경고 이력 또는 통계를 검토한다.

### 11.2 중복 예약 판단 기준 모호성

중복 예약은 단순한 `UNIQUE` 제약조건만으로는 충분하지 않다.

다음 방식이 필요하다.

1. 애플리케이션에서 저장 전 중복 시간 조회
2. SQLite 트리거로 최종 중복 차단
3. 동일 회의실, 동일 날짜, ACTIVE 상태만 중복 판단 대상으로 사용
4. 취소 상태 예약 제외 처리

## 12. 개발팀에게

이 제품은 복잡한 전사 회의실 플랫폼을 만드는 것이 아니다.

현재 목표는 17층 회의실 1, 회의실 2의 예약을 수기 대시보드로 관리하는 불편을 줄이고, 사용자가 예약 현황을 직접 확인하고 셀프로 예약할 수 있게 하는 것이다.

1차 구현에서는 다음 원칙을 우선한다.

1. 예약 현황을 한눈에 볼 수 있어야 한다.
2. 예약 등록이 수기 대시보드보다 쉬워야 한다.
3. 같은 회의실의 중복 예약은 반드시 막아야 한다.
4. 회의실 정원 초과는 차단하지 않고 경고 후 허용한다.
5. 예약은 1분 단위로 허용한다.
6. 1회 예약은 최대 2시간으로 제한한다.
7. 예약 변경과 취소가 단순해야 한다.
8. Windows 환경에서 실행 가능해야 한다.
9. SQLite에 예약 데이터를 저장한다.
10. 단일 PC 사용을 전제로 한다.
11. 관리자 기능은 1차 범위에서 제외한다.
12. 외부 인터넷 연결 없이 기본 기능이 동작해야 한다.

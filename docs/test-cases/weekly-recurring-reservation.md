# 주간 반복 예약 구현 테스트 케이스

| 항목 | 내용 |
|---|---|
| 대상 기능 | 주간 반복 예약 생성, 미리보기, 목록 표시, 일괄 수정/취소 확인 |
| 기준일 | 2026-06-12 |
| 기준 Feature | `features/reservation-weekly-recurring.feature` |
| 기준 UI Feature | `features/vue-weekly-recurring-reservation.feature` |
| 참고 SRS | `docs/SRS-v0.4-weekly-recurring-baseline.md` |
| 목적 | Gemini 구현의 완료 조건과 회귀 범위를 구현 전에 고정한다. |

## 1. 적용 원칙

1. API 경로, 요청 DTO, 응답 DTO는 기존 코드 또는 구현 시 확정된 명세를 사용한다.
2. 아래 TC는 특정 API 경로나 내부 테이블 구조를 가정하지 않는다.
3. 반복 예약 생성 범위는 시작일부터 종료월 마지막 날까지다.
4. 반복 예약 각 회차에는 기존 단건 예약 검증 정책을 동일하게 적용한다.
5. 일부 날짜가 충돌하면 먼저 충돌 날짜를 보여주고, 사용자 확인 후 가능한 날짜만 생성한다.
6. 미리보기는 데이터를 저장하지 않는다.
7. `P0`은 구현 완료 전에 반드시 통과해야 한다.
8. `P1`은 경계값과 주요 회귀 검증이다.

## 2. 공통 테스트 데이터

| 항목 | 값 |
|---|---|
| 회의실 1 ID | `ROOM_1` |
| 회의실 1 권고 정원 | 6명 |
| 회의실 2 ID | `ROOM_2` |
| 회의실 2 권고 정원 | 12명 |
| 운영시간 | `08:00~20:00` |
| 최대 예약 시간 | 2시간 |
| 기본 반복 제목 | `주간회의` |
| 기본 시작일 | `2026-06-15` 월요일 |
| 기본 종료월 | `2026-07` |
| 기본 시간 | `09:00~10:00` |
| 기본 예약자명 | `김철수` |
| 기본 참석 인원 | 5명 |
| 기본 회의 목적 | `주간 정례회의` |
| 기본 연락처 | `010-1234-5678` |

기본 반복 날짜:

```text
2026-06-15
2026-06-22
2026-06-29
2026-07-06
2026-07-13
2026-07-20
2026-07-27
```

## 3. 백엔드 및 도메인 TC

### TC-REC-001 정상 주간 반복 예약 생성

- 우선순위: `P0`
- 추적 요구사항: `REQ-REC-201-create-weekly`
- 사전조건: 회의실 1의 기본 반복 날짜와 시간에 ACTIVE 예약이 없다.
- 입력: 공통 기본값으로 주간 반복 예약을 생성한다.
- 기대 결과:
  - 생성 요청이 성공한다.
  - 서로 다른 예약 ID를 가진 ACTIVE 회차 7건이 생성된다.
  - 생성 날짜가 공통 기본 반복 날짜 7개와 정확히 일치한다.
  - 모든 회차의 회의실, 시간, 예약자명, 참석 인원, 목적, 연락처가 입력값과 일치한다.
  - 모든 회차가 동일한 반복 예약 그룹과 반복 제목 `주간회의`에 연결된다.

### TC-REC-002 시작일 요일을 반복 요일 기본값으로 사용

- 우선순위: `P0`
- 추적 요구사항: `REQ-REC-201-default-weekday`
- 입력: 시작일을 `2026-06-17`로 선택한다.
- 기대 결과:
  - 반복 요일 기본값이 수요일로 계산된다.
  - 사용자가 별도로 다른 요일을 확정하지 않은 상태에서는 수요일 날짜만 생성 대상이 된다.

### TC-REC-003 예상 생성 건수 미리보기

- 우선순위: `P0`
- 추적 요구사항: `REQ-REC-200-preview-count`
- 입력: 시작일 `2026-06-15`, 종료월 `2026-07`, 반복 요일 월요일로 미리보기를 요청한다.
- 기대 결과:
  - 예상 생성 건수가 7건이다.
  - 미리보기 실행 전후 저장된 예약 수가 변하지 않는다.

### TC-REC-004 생성 예정 날짜 미리보기

- 우선순위: `P0`
- 추적 요구사항: `REQ-REC-200-preview-occurrences`
- 입력: TC-REC-003과 같다.

  - 종료월 이후인 `2026-08-03`은 포함되지 않는다.
  - 미리보기 실행 전후 저장된 예약 수가 변하지 않는다.

### TC-REC-005 일부 충돌 날짜 미리보기

- 우선순위: `P0`
- 추적 요구사항: `REQ-REC-200-preview-conflicts`
- 사전조건: 회의실 1에 `2026-06-29 09:30~10:30` ACTIVE 예약이 있다.
- 입력: 기본 반복 예약을 미리보기 한다.
- 기대 결과:
  - 전체 생성 대상은 7건이다.
  - 충돌 날짜 목록에는 `2026-06-29`만 포함된다.
  - 가능한 날짜 수는 6건이다.
  - 미리보기 때문에 예약이 생성되지 않는다.

### TC-REC-006 사용자 확인 후 가능한 날짜만 생성

- 우선순위: `P0`
- 추적 요구사항: `REQ-REC-201-create-available-only`
- 사전조건: TC-REC-005와 같다.
- 절차:
  1. 기본 반복 예약을 미리보기 한다.
  2. 충돌 날짜를 확인한다.
  3. 가능한 날짜만 생성하는 작업을 확정한다.
- 기대 결과:
  - 생성 요청이 성공한다.
  - `2026-06-29`를 제외한 6건만 생성된다.
  - 기존 충돌 예약은 변경되지 않는다.
  - 중복 생성이나 부분 저장 오류가 없다.

### TC-UI-REC-001 반복 예약 체크박스 표시

- 우선순위: `P0`
- 추적 요구사항: `REQ-UI-REC-001-checkbox`
- 절차: 대시보드에서 예약 등록 화면을 연다.
- 기대 결과:
  - `주간 반복 예약` 체크박스가 표시된다.
  - 초기 상태는 단건 예약이며 기존 단건 예약 입력과 등록이 가능하다.

## 7. Feature 보강 전 구현 보류 항목

아래 항목은 SRS/PRD에 방향만 있고 Feature의 성공 인수조건이 충분하지 않다. Gemini가 임의로 정책이나 API를 정하지 않는다.

| 항목 | 현재 확인된 내용 | 추가로 확정할 내용 |
|---|---|---|
| 미래 회차 일괄 수정 성공 | 선택 회차 이후 모두 수정 | 수정 가능한 필드, 충돌 회차 처리, 트랜잭션 정책, 응답 결과 |
| 미래 회차 일괄 취소 성공 | 선택 회차 이후 모두 취소 | 이미 취소된 회차 처리, 과거 회차 제외 기준, 응답 결과 |
| 1개 회차만 수정 | 후보 기능 | MVP 포함 여부 |
| 1개 회차만 취소 | 후보 기능 | MVP 포함 여부 |
| 과거 회차 수정/취소 | 금지 추천 | 정확한 기준 시각과 오류 코드 |
| 반복 요일 직접 변경 | 시작일 요일을 기본값으로 사용 | 시작일과 다른 요일 입력 허용 여부 |
| 반복 그룹 저장 구조 | 그룹 식별자 연결 추천 | 실제 스키마와 마이그레이션 방식 |

## 8. Gemini 구현 완료 판정

구현 완료로 판정하려면 다음 조건을 모두 만족해야 한다.

1. `P0` TC가 모두 통과한다.
2. 실패 TC에서 부분 저장이 발생하지 않는다.
3. 미리보기 요청은 예약 데이터를 변경하지 않는다.
4. 반복 예약 추가 후 기존 생성, 목록, 수정, 취소 BDD가 모두 통과한다.
5. 프론트엔드 빌드가 통과한다.
6. Feature 보강 전 구현 보류 항목을 임의로 구현하지 않는다.

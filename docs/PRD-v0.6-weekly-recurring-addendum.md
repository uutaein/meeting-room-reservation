## 1. 배경

현재 시스템은 단건 예약을 중심으로 동작한다.

하지만 실제 업무에서는 매주 같은 요일, 같은 시간, 같은 회의실을 반복해서 사용하는 회의가 있다.

예시는 다음과 같다.

1. 매주 월요일 09:00~10:00 주간회의
2. 매주 수요일 14:00~15:00 프로젝트 정례회의
3. 매주 금요일 16:00~17:00 점검회의

이런 예약을 매번 수동으로 등록하면 사용성이 떨어지고, 누락 또는 중복 등록 가능성이 높아진다.

또한 프로젝트와 업무 일정은 보통 월 단위로 계획된다. 사용자는 “몇 회 반복할지”보다 “몇 월까지 반복할지”를 더 쉽게 알고 있다.

> 따라서 v0.6에서는 **주간 반복 예약**을 신규 요건으로 추가하고, 반복 종료 조건은 **반복 종료월** 기준으로 정의한다.

---

## 2. 목표

주간 반복 예약의 목표는 다음과 같다.

1. 사용자가 같은 회의실, 같은 요일, 같은 시간의 회의를 한 번에 등록할 수 있게 한다.
2. 반복 예약 생성 시 기존 단건 예약 정책을 그대로 적용한다.
3. 반복 종료 조건은 사용자가 이해하기 쉬운 `YYYY-MM` 종료월 기준으로 입력한다.
4. 반복 예약 생성 전 실제 생성 예정 날짜와 건수를 보여준다.
5. 반복 예약 수정 또는 취소 시 실수로 여러 예약이 바뀌지 않도록 강한 확인 절차를 둔다.
6. 반복 예약과 단건 예약이 같은 예약 현황 화면에서 자연스럽게 보이게 한다.
7. MVP에서는 복잡한 반복 규칙이 아니라 **매주 반복**만 지원한다.

---

## 3. 기능 개요

### 3.1 주간 반복 예약 생성

예약 생성 화면에 **주간 반복 예약** 선택 옵션을 추가한다.

MVP UI 방향은 다음과 같다.

1. 기본값은 단건 예약이다.
2. 사용자가 체크박스를 선택하면 주간 반복 예약 입력 영역을 표시한다.
3. 반복 요일은 예약 시작일의 요일을 기본값으로 사용한다.
4. 사용자는 반복 종료월을 `YYYY-MM` 형식으로 입력한다.
5. 시스템은 반복 시작일부터 반복 종료월의 마지막 날까지 해당 요일의 예약 날짜를 계산한다.
6. 시스템은 반복 기간 안의 각 날짜에 예약 회차를 생성한다.

예시 입력값은 다음과 같다.

| 항목 | 예시 |
|---|---|
| 회의실 | 회의실 1 |
| 예약 시작일 | 2026-06-15 |
| 반복 요일 | 월요일 |
| 시작 시간 | 09:00 |
| 종료 시간 | 10:00 |
| 반복 종료월 | 2026-07 |
| 예약 제목 | 주간회의 |
| 참석 인원 | 7 |
| 연락처 | 010-1234-5678 |

위 예시는 2026년 7월의 마지막 날까지 매주 월요일 예약을 생성하므로 다음 날짜가 생성 대상이다.

1. 2026-06-15 월요일
2. 2026-06-22 월요일
3. 2026-06-29 월요일
4. 2026-07-06 월요일
5. 2026-07-13 월요일
6. 2026-07-20 월요일
7. 2026-07-27 월요일

## 4. 반복 예약 정책

### 4.1 지원 범위

MVP에서 지원하는 반복 규칙은 다음으로 제한한다.

| 항목 | 정책 |
|---|---|
| 반복 주기 | 매주 |
| 반복 요일 | 1개 요일 |
| 반복 시간 | 동일 시작 시간, 동일 종료 시간 |
| 반복 회의실 | 동일 회의실 |
| 반복 종료 조건 | 반복 종료월 |
| 반복 종료월 형식 | `YYYY-MM` |
| 반복 종료 기준 | 해당 월의 마지막 날까지 |
| 반복 예외일 | MVP 제외 |
| 격주 반복 | MVP 제외 |
| 매월 반복 | MVP 제외 |
| 여러 요일 반복 | MVP 제외 |
| 반복 횟수 입력 | MVP 제외 |

### 4.2 단건 예약 정책과의 관계

주간 반복 예약도 기존 단건 예약 정책을 따른다.

| 기존 정책 | 반복 예약 적용 여부 |
|---|---|
| 1분 단위 예약 허용 | 적용 |
| 최대 2시간 제한 | 적용 |
| 운영시간 08:00~20:00 | 적용 |
| 같은 회의실 ACTIVE 예약 시간 겹침 방지 | 적용 |
| 정원 초과 경고 후 허용 | 적용 |
| 취소 예약 시간대 재예약 허용 | 적용 |

## 5. 충돌 처리 방향

반복 예약은 여러 날짜에 예약을 생성하므로, 일부 날짜에서 기존 예약과 시간이 겹칠 수 있다.

v0.6 PRD 추천안은 다음이다.

> 반복 예약 생성 시 충돌 날짜 목록을 먼저 보여주고, 사용자가 확인한 뒤 가능한 날짜만 생성한다.

사용자는 반복 종료월을 기준으로 생성 예정 날짜 전체를 확인한 뒤 예약을 진행해야 한다.

---

## 6. 수정 및 취소 확인 정책

반복 예약은 여러 예약을 동시에 바꿀 수 있으므로 일반 단건 예약보다 강한 확인 절차가 필요하다.

### 6.1 반복 예약 제목 확인

반복 예약 수정 또는 취소 시 사용자는 반복 예약 제목을 정확히 입력해야 한다.

방식은 GitHub 저장소 삭제 확인 UX와 유사하다.

| 항목 | 예시 |
|---|---|
| 반복 예약 제목 | 주간회의 |
| 안내 문구 | 이 반복 예약을 수정하려면 `주간회의`를 입력하세요. |
| 사용자 입력 | 주간회의 |
| 결과 | 입력값이 정확히 일치하면 수정 또는 취소 진행 |

### 6.2 확인이 필요한 작업

다음 작업은 제목 확인을 요구한다.

1. 반복 예약 전체 수정
2. 반복 예약 전체 취소
3. 반복 예약의 미래 일정 일괄 수정
4. 반복 예약의 미래 일정 일괄 취소

---

## 7. 반복 예약 수정 범위

반복 예약 수정 시 사용자는 수정 범위를 선택할 수 있어야 한다.

v0.6 PRD 추천안은 다음이다.

1. MVP에서는 **이 날짜 이후 모두 수정**을 우선 검토한다.
2. 과거 예약까지 바꾸는 전체 수정은 혼선을 줄 수 있으므로 후순위로 둔다.
3. 반복 예약 전체 수정이 필요하면 제목 확인을 반드시 요구한다.

---

## 8. 반복 예약 취소 범위

반복 예약 취소 시 사용자는 취소 범위를 선택할 수 있어야 한다.

v0.6 PRD 추천안은 다음이다.

1. MVP에서는 **이 날짜 이후 모두 취소**를 우선 검토한다.
2. 전체 취소는 제목 확인을 반드시 요구한다.
3. 과거 예약을 취소 상태로 바꿀 필요가 있는지는 SRS 단계에서 확정한다.

---

## 9. 화면 방향

### 9.1 예약 생성 화면

예약 생성 화면에는 다음 항목을 추가한다.

1. `주간 반복 예약` 체크박스
2. 반복 요일 표시
3. 반복 종료월 입력
4. 반복 생성 예상 건수 표시
5. 생성 예정 날짜 목록 표시
6. 충돌 예상 날짜 표시
7. 정원 초과 경고 표시

### 9.2 예약 목록 화면

반복 예약으로 생성된 예약은 목록에서 구분되어야 한다.

표시 방향은 다음과 같다.

1. 반복 예약 배지 표시
2. 반복 예약 제목 표시
3. 반복 그룹 ID 또는 반복 묶음 정보는 사용자에게 직접 노출하지 않아도 됨
4. 수정/취소 시 반복 예약 범위 선택 모달 표시

### 9.3 수정/취소 확인 화면

반복 예약 전체 또는 이후 일정을 수정/취소할 때는 다음을 표시한다.

1. 반복 예약 제목
2. 영향받는 예약 건수
3. 영향받는 날짜 범위
4. 확인 입력창
5. 취소 버튼
6. 진행 버튼

## 10. 데이터 방향

PRD 수준의 데이터 방향은 다음과 같다.

| 개념 | 설명 |
|---|---|
| 반복 예약 그룹 | 같은 반복 규칙으로 생성된 예약 묶음 |
| 반복 예약 제목 | 수정/취소 확인에 사용하는 사용자 입력 제목 |
| 반복 요일 | 매주 반복되는 요일 |
| 반복 시작일 | 반복 예약이 시작되는 날짜 |
| 반복 종료월 | 반복 예약이 끝나는 월 |
| 반복 회차 예약 | 실제 날짜별로 생성된 개별 예약 |

v0.6 PRD 추천안은 다음이다.

> MVP에서는 반복 예약 생성 시 날짜별 개별 예약을 미리 생성하고, 각 예약에 반복 그룹 식별자를 연결하는 방식을 우선 검토한다.

## 11. 성공 기준

주간 반복 예약 기능의 성공 기준은 다음과 같다.

1. 사용자는 체크박스로 주간 반복 예약을 선택할 수 있다.
2. 사용자는 반복 종료월을 `YYYY-MM` 형식으로 입력할 수 있다.
3. 시스템은 반복 시작일부터 반복 종료월 마지막 날까지 해당 요일의 예약 날짜를 계산할 수 있다.
4. 시스템은 각 날짜별 시간 겹침 여부를 확인할 수 있다.
5. 사용자는 반복 예약 생성 전 예상 생성 건수와 실제 생성 예정 날짜를 확인할 수 있다.
6. 사용자는 반복 예약 생성 전 충돌 날짜를 확인할 수 있다.
7. 반복 예약 전체 또는 이후 일정 수정/취소 시 제목 확인을 요구한다.
8. 사용자가 제목을 정확히 입력하지 않으면 수정/취소를 진행하지 않는다.
